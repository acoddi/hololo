# 프로젝트 개요

## 1. 프로젝트 정의

STM32 기반 스마트 식물 자동 관리 및 PC 모니터링 시스템

이 프로젝트는 센서 데이터를 읽고, 조건에 따라 장비를 제어하며, 현재 상태를 PC 애플리케이션에서 실시간으로 보여주는 시스템을 목표로 한다.

## 2. 핵심 목표

- 식물 관리에 필요한 센서 데이터를 주기적으로 수집한다.
- 토양 수분, 수위 등 상태에 따라 펌프와 팬을 제어한다.
- PC 애플리케이션에서 현재 상태를 실시간으로 모니터링한다.
- 자동 모드와 수동 모드를 모두 지원한다.
- 이벤트, 경고, 제어 이력을 로그로 저장한다.

## 3. 전체 구조

시스템은 크게 4개의 계층으로 나눈다.

### 3.1 Device Layer

실제 하드웨어를 다루는 계층이다.

- STM32
- 토양 수분 센서
- 온습도 센서
- 수위 센서
- 유량 센서
- 릴레이
- 펌프
- 팬

역할:

- 센서 데이터 측정
- 액추에이터 제어
- 기본 안전 제어 수행

### 3.2 Communication Layer

STM32와 PC를 연결하는 계층이다.

- UART 기반 시리얼 통신
- 양방향 송수신
- 데이터 패킷 규약 관리

역할:

- STM32 -> PC: 센서값, 상태값, 경고 전송
- PC -> STM32: 제어 명령, 모드 변경, 임계치 설정 전송

### 3.3 Application Logic Layer

PC 프로그램 내부 로직 계층이다.

- 시리얼 데이터 수신 및 파싱
- 현재 상태 관리
- 자동/수동 제어 로직
- 임계치 판단
- 로그 저장
- 알람 처리

즉, 시스템의 판단과 상태 관리를 담당하는 계층이다.

### 3.4 Presentation Layer

사용자가 직접 보는 화면 계층이다.

- Qt + QML UI
- 실시간 센서 대시보드
- 제어 버튼
- 연결 상태 표시
- 로그 조회
- 경고 표시

즉, 사용자가 시스템을 이해하고 조작하는 관제 화면이다.

## 4. 전체 아키텍처

```text
[센서/장비]
  └─ 토양수분 / 온습도 / 수위 / 유량 / 펌프 / 팬
        ↓
     [STM32]
  - 센서값 수집
  - 기본 자동 제어
  - 명령 수신
  - 상태 전송
        ↓ UART
  =========================
        ↑↓ 양방향 통신
  =========================
        ↓
     [PC Application]
  ├─ SerialManager
  ├─ SensorDataParser
  ├─ DeviceService
  ├─ AutomationService
  ├─ LogService
  ├─ DatabaseManager
  ├─ ViewModel
  └─ QML UI
        ↓
      [사용자]
```

## 5. 주요 데이터 흐름

### 5.1 센서 데이터 수집 흐름

센서 -> STM32 -> UART 송신 -> PC 수신 -> 파싱 -> 상태 갱신 -> UI 반영 -> 로그 저장

예시:

1. STM32가 토양 수분 28%를 읽는다.
2. UART로 센서 패킷을 전송한다.
3. PC의 `SerialManager`가 데이터를 수신한다.
4. `SensorDataParser`가 패킷을 구조화된 데이터로 변환한다.
5. ViewModel이 `soilMoisture` 값을 갱신한다.
6. QML UI 카드가 자동으로 다시 그려진다.
7. 필요 시 경고와 로그가 기록된다.

### 5.2 사용자 제어 흐름

사용자 입력 -> QML -> ViewModel -> Service -> SerialManager -> STM32 -> 제어 결과 회신 -> UI 갱신

예시:

1. 사용자가 펌프 ON 버튼을 클릭한다.
2. QML이 `ControlViewModel::startPump()`를 호출한다.
3. `DeviceService`가 `CMD,PUMP,ON` 명령을 생성한다.
4. `SerialManager`가 명령을 STM32로 전송한다.
5. STM32가 릴레이를 동작시킨다.
6. STM32가 `ACK,PUMP,ON`을 회신한다.
7. PC가 장비 상태를 갱신하고 UI에 반영한다.

### 5.3 자동 제어 흐름

자동 제어는 아래 두 방식 중 하나 또는 혼합형으로 운영한다.

방식 1: STM32 중심 자동 제어

- 장점: PC가 없어도 동작 가능하다.
- 용도: 최소 생존 로직, 안전 동작, 긴급 대응

방식 2: PC 중심 자동 제어

- 장점: 로직 변경, 설정 연동, 시각화가 쉽다.
- 용도: 고급 조건 제어, 기록 기반 판단, 사용자 설정 반영

권장 방식: 혼합형

- STM32는 최소 생존 로직과 안전 제어 담당
- PC는 관제, 설정, 로그, 고급 자동 제어 담당

## 6. 통신 프로토콜 초안

초기 구현은 문자열 기반 프로토콜을 사용한다.

### 6.1 STM32 -> PC

```text
SENSOR,TEMP=24.1,HUM=51,SOIL=32,WATER=78,FLOW=12
STATUS,PUMP=OFF,FAN=ON,MODE=AUTO
ALERT,WATER_LOW
```

### 6.2 PC -> STM32

```text
CMD,PUMP,ON
CMD,PUMP,OFF
CMD,FAN,ON
CMD,MODE,AUTO
CMD,MODE,MANUAL
CMD,THRESHOLD,SOIL,35
```

### 6.3 STM32 응답

```text
ACK,PUMP,ON
ACK,MODE,AUTO
ERR,INVALID_COMMAND
```

프로토콜 설계 원칙:

- 사람이 읽을 수 있어야 한다.
- UART 로그만 봐도 상태 추적이 가능해야 한다.
- 패킷 종류는 첫 토큰으로 구분한다.
- 값은 `KEY=VALUE` 형식을 기본으로 사용한다.
- 잘못된 명령에 대해서는 반드시 오류 응답을 준다.

## 9. 핵심 설계 포인트

- UI와 로직을 분리한다.
- 하드웨어 제어와 PC 애플리케이션 책임을 분리한다.
- 센서 모니터링뿐 아니라 양방향 제어 구조를 갖는다.
- 연결 상태, 장비 상태, 경고 상태를 명확히 관리한다.
- 이후 RF, Wi-Fi, MQTT 같은 확장 경로를 고려한다.

## 10. 한 줄 요약 아키텍처

STM32가 센서 측정과 장비 제어를 담당하고, PC의 Qt/QML 애플리케이션이 UART 기반으로 데이터를 수신해 실시간 모니터링, 수동/자동 제어, 로그 저장, 설정 관리를 수행하는 구조

## 11. 권장 개발 순서

1. STM32에서 가짜 센서 데이터 송신 기능 구현
2. Qt에서 `QSerialPort` 기반 수신 구현
3. QML 대시보드에 센서값 표시
4. 버튼으로 STM32 제어 명령 전송
5. 토양 수분 기준 자동 급수 로직 추가
6. SQLite 로그 저장 기능 추가
7. 설정 화면 추가
8. 경고 및 알람 UI 추가
